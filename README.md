# CS 230 Portfolio: The Gaming Room Software Design

## Client and Software Requirements

The client was **The Gaming Room**, a company that created the game *Draw It or Lose It*. They wanted to expand the game from Android to a web-based application that could support multiple teams and players. The software needed to make sure game names and team names were unique, allow many users to play at the same time, and work on different operating systems.

## What I Did Well

I think I did a good job explaining the software requirements and creating the system design. I also completed the UML class diagram and compared different operating systems before making my recommendation. This helped show how the application should be built and maintained.

## Helpful Parts of the Design Process

Working through the design document helped me understand the project before writing code. It allowed me to organize the requirements, think about the system architecture, and plan how the classes would work together. This made the development process easier.

## What I Would Improve

If I could revise one part of the document, I would add more details about security and system scalability. I would also improve the UML diagram by adding more explanations so it is easier to understand for other developers.

## Understanding the User's Needs

I focused on the client's requirements and designed the software to support multiple users, unique game and team names, and future growth. I recommended using object-oriented programming, a client-server architecture, and Linux because they support reliability, performance, and future expansion. Understanding the user's needs is important because the software must solve the client's problem and provide a good experience for the people using it.

## My Software Design Approach

I approached this project by reading the requirements carefully, breaking the project into smaller parts, and designing the system before thinking about implementation. In future projects, I will continue using UML diagrams, software design documents, and object-oriented design principles because they help organize ideas and reduce problems during development.
